## 一、领域对象如何被消费

在本次实现中，View 层并不直接操作 Sudoku 或 Game 对象，而是通过一个中间适配层 `gameStore` 来消费领域对象。该 store 内部持有一个 Game 实例，并对外暴露响应式数据与操作接口。

View 层直接消费的是 `gameStore`，通过 Svelte 的 `$gameStore` 语法获取当前状态，例如 `grid`、`canUndo`、`canRedo` 等。组件中的棋盘渲染完全依赖 `$gameStore.grid`，而不再使用原有的 `userGrid` 或直接操作二维数组。

用户操作通过事件进入领域对象，例如在键盘输入时调用 `gameStore.guess(x, y, value)`，撤销和重做通过 `gameStore.undo()` 与 `gameStore.redo()` 完成。这些方法内部再调用 Game 和 Sudoku 的逻辑，从而保证所有核心行为都由领域对象驱动。

当领域对象状态发生变化时，store 会通过 `set()` 或 `update()` 触发新的对象引用，从而让 Svelte 检测到变化并更新 UI，因此界面能够自动刷新。

---

## 二、响应式机制说明

本设计主要依赖 Svelte 的 store 机制以及 `$store` 自动订阅能力。`gameStore` 是一个自定义 store，对外暴露 subscribe 方法，因此可以被 `$gameStore` 使用。

UI 中的响应式更新来源于以下机制：

* 使用 `$gameStore` 自动订阅状态
* 每次状态变化时，通过重新赋值（而非直接修改内部字段）触发更新

例如，在执行 guess 或 undo 操作后，会生成新的 grid 或新的 Game 状态，并通过 store 更新，这样 Svelte 能检测到引用变化并刷新组件。

在本方案中：

* `grid`、`canUndo`、`canRedo` 属于对 UI 可见的响应式状态
* 历史记录、内部校验逻辑等仍封装在 Game / Sudoku 内部，不直接暴露

如果直接修改对象内部字段（例如 `grid[row][col] = value`），由于引用未变化，Svelte 不会触发更新，从而导致界面不刷新。因此本设计避免直接 mutate，而是通过新的状态对象进行更新。

## 三、领域对象改进说明

相比 Homework 1，本次设计做了以下改进：

首先，明确了职责边界。Sudoku 负责维护棋盘数据、校验合法性以及提供基本操作接口，而 Game 负责管理历史（Undo / Redo）以及对外提供统一操作入口。UI 不再直接操作 Sudoku，而是统一通过 Game 进行交互。

其次，改进了历史管理方式。Game 内部维护状态栈，在每次操作时保存快照，从而支持 undo 和 redo。这种方式使状态回溯更加清晰，也避免了 UI 层自行维护历史的问题。

再次，引入了 store adapter（gameStore），作为领域对象与 Svelte 之间的桥梁。这一层将领域对象转换为响应式数据，并统一对外暴露接口，使 UI 不再依赖具体实现细节。

在 HW1 中，领域对象虽然存在，但没有真正接入 UI，导致组件仍直接操作旧状态，违背了设计初衷。本次通过 store 层的引入，解决了这一问题，使领域对象成为真正的核心。

当然，这种设计也存在一定 trade-off，例如增加了一层抽象，代码结构更复杂，但换来了更清晰的职责划分与更好的扩展性。

---

## 四、Svelte 与领域对象协作方式

本设计中，Svelte 与领域对象的协作通过 store 完成：

* 领域对象（Game / Sudoku）负责业务逻辑
* store 负责响应式包装
* View 只负责展示与事件触发

这种结构保证了 UI 与业务逻辑解耦，同时利用 Svelte 的响应式机制实现自动更新。

当用户进行输入或操作时，事件会调用 store 方法，store 再调用领域对象更新状态，并通过重新赋值触发订阅更新，最终驱动 UI 重新渲染。

---

## 五、总结

本次实现的核心在于将领域对象真正接入 Svelte 应用流程，通过引入 `gameStore` 作为适配层，实现了：

* UI 不再直接操作数据，而是通过领域对象接口
* Undo / Redo 完全由 Game 管理
* 状态更新能够正确触发响应式刷新

该设计不仅满足了作业要求，也为后续扩展（如提示、探索模式）提供了良好的基础。
